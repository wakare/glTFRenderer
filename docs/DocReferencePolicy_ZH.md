# 文档引用规范

对应英文版：

- `docs/DocReferencePolicy.md`

## 1. 目的

在持续重构阶段，尽量让架构/算法文档保持可信，同时控制维护成本。

## 2. 路径基准

所有 repo 相对路径都以仓库根目录为基准。

- 源码与解决方案树使用 `glTFRenderer/...`
- 顶层脚本与文档使用 `scripts/...`、`docs/...`
- 只有在描述可执行程序运行时工作目录行为时，才使用 `Resources/...` 这类运行时相对路径

## 3. 首选引用风格

优先使用：

- `路径 + 符号名`

例如：

- `glTFRenderer/RendererCore/Private/RendererInterface.cpp`
- 文本中说明符号名 `ExecuteRenderGraphFrame`

只有在行号上下文确实重要时，才加：

- `path:line`
- `path#Lline`

## 4. 什么时候必须更新引用

以下情况必须在同一改动里更新文档引用：

1. 被引用文件移动、重命名或删除
2. 被引用符号改名或移动
3. 行号引用已经失效

## 5. 双语与说明图规则

对 `docs/` 下项目文档：

1. 新建或实质更新的文档，应同时维护中文和英文覆盖
2. 已存在的无后缀文档可保留，另一语言版本放在同目录下，命名为 `_EN.md` 或 `_ZH.md`
3. 渲染架构和应用层框架文档应包含关键类依赖说明图
4. 说明图应体现：
   - 关键类依赖方向
   - 每个关键类的核心职责
5. 当文档主体主要是图时，可以使用一份双语图源作为共享说明图

推荐图格式：

- `docs/` 下的 Mermaid markdown 文档

## 6. 校验命令

提交前运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Validate-DocReferences.ps1
```

脚本行为：

- 扫描 `docs/**/*.md`
- 校验反引号中的文件路径引用
- 任一错误都会返回非零退出码

## 7. PR 检查建议

- [ ] 已运行文档引用校验
- [ ] 文档陈述已重新对照当前代码行为
- [ ] 若为新文档或实质更新，已同步更新中英文覆盖
- [ ] 若为渲染架构文档，已更新或复核说明图

## 8. 轻量正确性审查

对架构/算法文档，合并前至少做以下检查：

1. 逐节找出关键陈述
2. 用当前代码实现重新核对
3. 如果行为已变化，在同一改动中更新文档和引用
